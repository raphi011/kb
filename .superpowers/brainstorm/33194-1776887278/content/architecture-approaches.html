Architecture Approaches

How to structure the core library, CLI, and server

A

Layered Library

Core logic in pkg/ packages: repo (go-git wrapper), index (SQLite indexer), search (FTS queries), markdown (parser/renderer). CLI and server are thin consumers.

Pros

  • Clean separation — CLI and server share identical logic
  • Testable in isolation (no HTTP, no CLI glue)
  • Future agent just imports the same packages

Cons

  • Requires disciplined API boundaries upfront
  • More packages to wire together initially
B

Service-Oriented

Single internal/kb service struct that wraps git + index. CLI and server both call methods on this service. All state flows through one type.

Pros

  • Simple to start — one struct, one package
  • Easy to understand the full API surface
  • Natural concurrency boundary (mutex on write ops)

Cons

  • Service struct may grow large over time
  • Harder to test subsets in isolation
C

Hybrid (Recommended)

Focused internal packages (gitrepo, index, markdown) composed by a thin kb.KB service. Packages are independently testable, but the service provides the unified API for CLI and server.

Pros

  • Best of A and B — clean packages + simple entry point
  • Each package has one job, service wires them together
  • Grows naturally — split the service only when needed

Cons

  • Slightly more structure than B upfront
