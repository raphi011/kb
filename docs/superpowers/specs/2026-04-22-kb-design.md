# kb — Git-backed Markdown Knowledge Base

A Go application that uses a Git repository as the source of truth for markdown files, indexes them in SQLite for fast search and navigation, and serves a web UI with a git remote.

## Goals

- **CLI tool**: index a local git repo, search/browse notes from the terminal with fzf support
- **Web server**: read-only web UI (ported from zk-serve), REST API, git smart HTTP remote
- **Simple infrastructure**: no external databases, no embedding pipelines — Git + SQLite + single binary
- **Incremental indexing**: only re-process files that changed since the last indexed commit

## Non-Goals (v1)

- Write API / edit UI (future)
- Agent / LLM integration (future)
- Multi-repo support (future — design supports adding later)
- User accounts / multi-user (future)
- Embedding-based semantic search (future)

### Package Responsibilities

**`gitrepo`** — Isolates all go-git operations. The rest of the codebase never imports go-git directly. Responsibilities:
- Open a `*git.Repository` from a filesystem path
- Walk the HEAD tree to list files
- Read blob content by path
- Diff two commits to get added/modified/deleted paths
- Build a path → (created, modified) timestamp map from `git log` in a single pass
- Serve git smart HTTP protocol (upload-pack for pulls, receive-pack for pushes)

**`index`** — Owns the SQLite database schema and all queries. Responsibilities:
- Create/migrate schema on open
- CRUD operations for notes, tags, links
- FTS5 search with BM25 ranking and tag filtering
- Aggregation queries: tags with counts, activity by day, backlinks
- Track last indexed commit SHA via `index_meta` table

**`markdown`** — Parses and renders markdown. Ported from knowhow's `internal/parser/`. Responsibilities:
- Parse YAML frontmatter → structured metadata
- Extract wiki-links: `[[target]]` syntax
- Extract inline tags: `#tag` patterns (not in code blocks)
- Extract outgoing links: `[text](url)` and autolinks
- Parse heading structure for table of contents
- Extract lead paragraph (first non-empty paragraph)
- Render markdown → HTML via Goldmark with syntax highlighting and wiki-link resolution

**`kb`** — The service layer. Composes the three packages above. Responsibilities:
- `Open(repoPath) → *KB` — opens git repo + SQLite index
- `Index()` — full or incremental index (auto-detects based on index_meta)
- `Search(query, tags) → []Note` — FTS search with optional tag filter
- `NoteByPath(path) → Note` — single note with parsed content
- `AllTags() → []Tag` — tags with note counts
- `OutgoingLinks(path) → []Link`
- `Backlinks(path) → []Link`
- `AllNotes() → []Note` — for sidebar tree
- `NotesByDate(date) → []Note` — for calendar
- `ActivityDays(year, month) → []int` — for calendar widget
- `ReadFile(path) → []byte` — raw markdown from git blob

**`server`** — HTTP server consuming the KB service. Responsibilities:
- Route registration and middleware (auth, logging)
- Web UI handlers with Templ rendering and HTMX partial support
- Content negotiation: same routes serve HTML, HTMX partials, or JSON
- Git smart HTTP endpoints
- Trigger incremental re-index after receiving a git push
- Static asset serving (embedded via `//go:embed`)

## SQLite Schema

```sql
CREATE TABLE notes (
    path        TEXT PRIMARY KEY,
    title       TEXT NOT NULL,
    body        TEXT NOT NULL,
    lead        TEXT,
    word_count  INTEGER NOT NULL,
    created     DATETIME,
    modified    DATETIME,
    metadata    TEXT
);

CREATE VIRTUAL TABLE notes_fts USING fts5(
    title, body, path,
    content='notes',
    content_rowid='rowid'
);

CREATE TABLE tags (
    name    TEXT NOT NULL,
    path    TEXT NOT NULL REFERENCES notes(path) ON DELETE CASCADE,
    PRIMARY KEY (name, path)
);

CREATE TABLE links (
    source_path TEXT NOT NULL REFERENCES notes(path) ON DELETE CASCADE,
    target_path TEXT NOT NULL,
    title       TEXT,
    external    BOOLEAN NOT NULL DEFAULT 0,
    PRIMARY KEY (source_path, target_path)
);

CREATE TABLE index_meta (
    key   TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
```

**Design notes:**
- `notes.body` stores raw markdown (used by FTS5 and for raw content serving)
- `notes.metadata` stores frontmatter as JSON (for display, not queried)
- `notes_fts` uses `content='notes'` so FTS reads from the notes table — no data duplication. Requires explicit triggers (or manual `INSERT INTO notes_fts` / `DELETE FROM notes_fts`) on notes CRUD to keep the FTS index in sync — the `index` package handles this
- Backlinks are a reverse query on `links`: `SELECT * FROM links WHERE target_path = ?`
- `index_meta` stores `("head_commit", "<sha>")` for incremental indexing
- `tags.name` is the normalized tag string; inline `#tags` and frontmatter `labels` both go here
- CASCADE delete on `notes.path` cleans up tags and links automatically

## Indexing

### Full Index (first run or `kb index --full`)

1. Open git repo via go-git
2. Single-pass `git log` to build path → (first_commit_date, last_commit_date) map
3. Walk HEAD tree — for each `.md` file:
   - Read blob content from git
   - Parse with markdown package → title, lead, tags, links, frontmatter, word count
   - Insert into `notes`, `tags`, `links` tables
4. Rebuild FTS5 index
5. Store HEAD commit SHA in `index_meta`

### Incremental Index (subsequent runs or post-push)

1. Read last indexed SHA from `index_meta`
2. Diff old_commit..HEAD via go-git → added, modified, deleted paths
3. Deleted `.md` files: `DELETE FROM notes WHERE path = ?` (CASCADE cleans tags/links)
4. Added/Modified `.md` files: parse and upsert
5. Update FTS5 for changed rows only
6. Store new HEAD SHA in `index_meta`

### Server-Side Trigger

After `git-receive-pack` completes (client pushed), the server automatically runs incremental indexing. The web UI reflects new content immediately.

### Performance

- `git log` for timestamps: single pass over all history, not per-file — O(commits) not O(files × commits)
- Incremental index only processes changed files — typical push touches few files
- SQLite WAL mode for concurrent reads during index writes

## CLI Commands

```
kb index [path]              # Index repo at path (default: current dir)
                             # Full on first run, incremental after
                             # Creates .kb.db next to .git/

kb search [-i] <query>       # FTS search, prints matching notes with snippets
    --tags tag1,tag2         # Filter by tags
    --limit 20               # Max results (default 20)
                             # -i: pipes to fzf with preview

kb list [-i]                 # List all notes
                             # -i: fzf with full-text filter + preview

kb tags                      # List all tags with note counts

kb links <path>              # Show outgoing links from a note
kb backlinks <path>          # Show notes that link to this path

kb cat <path>                # Print raw markdown of a note

kb edit -i                   # fzf select → open in $EDITOR

kb serve                     # Start web server + git remote
    --addr :8080             # Listen address (default :8080)
    --repo <path>            # Repo path (default: current dir)
    --token <secret>         # Auth token (required)
```

### fzf Integration

When `-i` is passed and fzf is available:
- Results formatted as `path\ttitle\ttags` for fzf input
- Preview command: `kb cat {path}` shows rendered note
- Selected note's path printed to stdout (composable: `vim $(kb search -i "git")`)

### Index Location

`.kb.db` is created next to `.git/` in the repo root. No configuration files for v1 — the repo path is the only input. CLI commands auto-detect the repo root by walking up from the current directory (like `git` itself).

## Server

### Routes

```
GET  /notes/{path...}       # Note or folder view
GET  /search?q=&tags=       # Search
GET  /tags                  # Tag list
GET  /calendar?year=&month= # Calendar widget
GET  /static/*              # Embedded assets
GET  /healthz               # Health check (unauthenticated)
GET  /login                 # Login form (unauthenticated)
POST /login                 # Login submit (unauthenticated)
GET  /                      # Redirect to /notes/

# Git smart HTTP
GET  /git/info/refs         # Reference discovery
POST /git/git-upload-pack   # Client pulls
POST /git/git-receive-pack  # Client pushes → triggers re-index
```

### Content Negotiation

Same routes serve different content types:
- **Browser request** → full HTML page (Templ layout + content)
- **HTMX request** (`HX-Request` header) → HTML partial (content + OOB TOC swap)
- **JSON request** (`Accept: application/json`) → JSON response (for API consumers, future agent)

### Authentication

Single shared token, required to start the server (`--token` flag).

| Client | Auth Method | Credential |
|--------|-------------|------------|
| Web UI | Session cookie | Login form → HTTP-only cookie signed with token |
| API / JSON | Bearer header | `Authorization: Bearer <token>` |
| Git (push/pull) | HTTP Basic Auth | Password = token (username ignored) |
| Health check | None | `/healthz` is unauthenticated |
| Login page | None | `/login` is unauthenticated |

No token = server refuses to start. CLI commands bypass auth (direct local DB access).

### Web UI (Ported from zk-serve)

Three-column layout:
- **Left**: sidebar with folder tree, tag list, search input, client-side filtering
- **Center**: note content (rendered markdown) or folder listing, with breadcrumbs
- **Right**: table of contents, outgoing links, backlinks, calendar widget

Features carried over:
- HTMX partial page updates (no full reload on navigation)
- Command palette (Cmd+K) with fuzzy search
- Tag filtering (multiple tags)
- Calendar widget with activity indicators
- Dark/light theme toggle with localStorage
- Zen mode (hide sidebar/TOC)
- Resizable panels
- Keyboard shortcuts
- Mermaid diagram rendering
- Syntax highlighting (Chroma)
- Wiki-link resolution in rendered markdown

Adapted from zk-serve:
- Handlers query `kb.KB` service instead of zk's `notebook.db`
- File content read from git blobs via KB service
- Routes use `/notes/{path}` prefix instead of `/note/{path}` and `/folder/{path}`
- Content negotiation added (JSON responses on same routes)
